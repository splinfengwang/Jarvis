# Topic Lifecycle

## 目的

Topic 是持续工作容器，不是所有问题都需要创建。Topic 规则负责让跨会话工作可恢复、可关闭、可萃取。

## 创建准入

满足任一条件时创建 Topic：

| 条件 | 说明 |
|---|---|
| 林峰明确要求 | “开一个 Topic”“先建立个 Topic” |
| 预计跨会话推进 | 当前工作无法在本轮自然结束 |
| 会产生可交付文件 | 方案、报告、PPT、评审、草案、清单 |
| 有外部依赖或跟进事项 | 有责任人、截止窗口、等待反馈 |
| 会形成决策链 | 后续需要追踪范围、取舍、风险、未解决问题 |
| 需要知识萃取 | 本轮内容未来需要入库或复用 |

不创建 Topic：单次问答、临时评价、简单状态确认、用户明确说只讨论/不写入/不建档。

## 标准骨架

```text
platform-ops/topics/YYYYMMDD_主题简称/
  ├── 索引.md
  ├── _上下文快照.md
  ├── _准入检查单.md
  ├── 讨论记录.md
  ├── 参考资料/          ← 外部输入材料（PDF/PPT/Word/图片/网页原文等），只读
  ├── 过程稿/            ← 迭代中间版本、被取代的旧版、废弃方案；萃取时默认跳过
  └── 定稿/              ← 讨论确认后的最终产出；萃取首选扫描目标
```

### 子目录规则

- 三个标准目录下**允许按子主题建立二级目录**（如 `定稿/服务需求管理/`、`过程稿/双水平无创呼吸机-患者款/`）。
- 单问题无子主题时，文件直接放在对应目录下，不强制建二级目录。
- 根目录仅保留四件套骨架文件和元数据文件（`_萃取清单.md`、`_evidence-pack.json` 等）。

### 文件命名规范

**核心原则：文件名应清晰概括文件内容，不能写成 `2026-06-08方案.md` 这种毫无信息量的名称。** 任何人看了文件名就应该知道这个文件讲的是什么。

**上下文原则：文件名必须能在脱离目录结构后仍被理解。** 一个文件从 Topic 目录提取到 `业务/` 后，文件名本身应该能告诉读者它讲什么、何时创建——不能依赖父目录来补全含义。

**日期原则：所有产出文件必须带日期前缀。** 同一 Topic 内不同文件可能在不同日期产生，日期记录产出时间点。Topic 目录日期不等于文件日期。

**分隔符约定：** 目录名使用 `_` 分隔日期与主题简称（`YYYYMMDD_主题简称`），文件名内部使用 `-` 分隔各部分（`YYYYMMDD-主题-文档类型.md`）。

#### 定稿/

- 格式：`<YYYYMMDD>-<主题>-<文档类型>.md`，如 `20260605-适老化改造-架构总览.md`
- **不加 `_` 前缀** — `_` 前缀仅用于骨架和元数据文件
- 子主题有多个产出时建二级目录，文件直接放目录下：
  ```
  定稿/
    ├── 20260605-适老化改造-架构总览.md
    ├── 20260605-适老化改造-服务需求管理BRD.md
    ├── 20260605-适老化改造-空间配置管理BRD.md
    ├── 20260605-适老化改造-安防配置管理BRD.md
    ├── 20260606-适老化改造-Roundtable审查报告.md
    └── 双水平无创呼吸机-患者款/
        ├── 20260518-双水平无创呼吸机-患者款-文案.md
        └── 20260520-双水平无创呼吸机-患者款-宣传页.pdf
  ```
- 二级子目录无例外 — 文件同样必须带 `日期-主题`，不能因为父目录名提供了上下文就只写 `文案.md`。萃取后文件可能脱离目录结构独立存在，文件名必须自包含全部信息。
- 旧版定稿被新版本取代时，移入 `过程稿/`；`定稿/` 始终保持最新版本。

- ✅ 好的命名：`20260605-适老化改造-架构总览.md`、`20260608-用户积分体系-设计方案.md`、`20260518-新松呼吸机彩页-竞品对比分析.md`
- ❌ 差的命名：`2026-06-08方案.md`（只有日期+笼统类型）、`架构总览.md`（无日期无主题）、`文档.md`、`v1.0.md`

#### 过程稿/

- 格式：`<YYYYMMDD>-<主题>-<文档类型>-v<N.M>.md`，如 `20260608-用户积分体系-设计方案-v0.1.md`
- 版本号从 0.1 起步，每次实质修改递增小版本（0.1 → 0.2 → 0.3...），重大重构跳大版本（1.0）
- 同一日期产出多版本是正常的（当天迭代），日期不变、版本号递增
- 跨日修改 → 日期更新，版本号从新日期继续
- 被取代的旧版保留，不删除：
  ```
  过程稿/
    ├── 20260608-用户积分体系-设计方案-v0.1.md
    ├── 20260608-用户积分体系-设计方案-v0.2.md
    └── 20260610-用户积分体系-设计方案-v0.3.md   ← 两天后新版本
  ```
- 用户修改或外部输入版本用描述性后缀而非版本号：`20260520-新松呼吸机彩页-文案-涛哥修改版.md`

- ✅ 好的命名：`20260608-用户积分体系-设计方案-v0.1.md`、`20260605-适老化改造-架构总览-v1.0.md`
- ❌ 差的命名：`新文档.md`、`方案最新.md`、`方案v2.md`（无日期无主题）、`方案(改).md`

#### 参考资料/

- 保持原始文件名或略加描述前缀
- 来源多时按类型建子目录：`竞品参考/`、`厂家素材/`、`官方制度/`
- 不强制改名，因为参考资料的价值在于可溯源

- ✅ 可接受的命名：`科研项目经费管理办法.md`（原文标题）、`参考彩页_BMC.md`（加描述前缀）、`产品需求原始笔记.md`（用户输入）

## Topic Capsule

吸收 ByteRover Context Tree 和 Obsidian Copilot Project Mode 思路。Topic Capsule 是可恢复状态对象，不是独立 skill。

最小字段：

```text
topic_id
status
scope
last_action
next_action
confirmed_facts
decisions
open_questions
risk_boundaries
key_files
context_pack
source_sessions
do_not_touch
```

限制：

- 不塞整个 vault。
- 不自动改写 confirmed_facts。
- 仪表盘是入口，状态事实以 Topic 本体为准。

## 讨论记录中的产物回链

`讨论记录.md` 不只记录聊了什么，还要在产生产物时显式记录“讨论/决策 → 产物”的映射。

### 规则

- 当本轮讨论形成了新的 `过程稿/` 或 `定稿/` 文件时，必须在 `讨论记录.md` 追加一条**产物生成事件**。
- 产物生成事件记录的是**关系和状态**，不是产物正文；**不得**把完整产物内容复制进 `讨论记录.md`。
- 同一产物若后续被新版本替代，新的产物生成事件中要写明替代关系；旧版仍保留在 `过程稿/`。
- 若产物随后迁移到 `业务/<域>/`，讨论记录仍保留原始 Topic 内链接或补充迁移说明，作为过程审计链。

### 最小字段

- 讨论主题
- 关键结论
- 产出动作（链接到具体文件）
- 产物状态（`过程稿` / `定稿`）
- 替代关系（如有）

### 推荐格式

```markdown
## 2026-06-26 产物生成

- 讨论主题：xxx
- 关键结论：yyy
- 产出动作：形成 [[过程稿/20260626-xxx-文档类型-v0.1.md|《xxx》]]
- 产物状态：过程稿
```

```markdown
## 2026-06-26 产物生成

- 讨论主题：xxx
- 关键结论：yyy
- 产出动作：记录为 [[定稿/20260626-xxx-文档类型.md|《xxx》]]
- 产物状态：定稿
- 替代关系：替代 [[过程稿/20260625-xxx-文档类型-v0.2.md|上一版]]
```

## 会话捕获

Topic 生命周期动作必须维护 `_上下文快照.md` 的 `## 6. 关联会话` 表。

必经时机：

| 动作 | 必须记录 |
|---|---|
| Topic 创建 | 当前会话行 |
| Topic 冻结 | 如果本轮会话未在表内，追加当前会话行 |
| Topic 关闭 | 追加最终会话行 |
| 知识萃取 | 优先使用关联会话表里的 JSONL 路径 |

执行规则：

1. 先按 `references/session-locating.md` 或 `locate_session_jsonl.py` 定位当前会话 JSONL。
2. 精确命中时写入 `工具 / 会话标识 / JSONL 路径 / 工作区路径 / 日期`。
3. 找不到精确 JSONL 时，仍写入工具、会话标识或 `待确认`、工作区路径和日期；JSONL 路径写 `待确认`。
4. 同一会话行已存在时不得重复追加。
5. 不允许把会话捕获留给后续萃取阶段补做。

## 状态值

| 机器值 | 展示值 | 含义 |
|---|---|---|
| `doing` | `[🟢 Doing]` | 当前焦点 |
| `paused` | `[🟡 Paused]` | 暂停，可恢复 |
| `blocked` | `[🔴 Blocked]` | 外部依赖阻塞 |
| `pending_extraction` | `[📋 待萃取]` | 工作完成，等待知识萃取 |
| `done` | `[⚪ Done]` | 萃取完成，归档 |

冲突裁决顺序：本轮明确状态变更 > `_上下文快照.md` > `索引.md` > 仪表盘。

## 冻结

触发：漂移切换、林峰主动暂停、需要保存现场。

必须同步：

1. 仪表盘状态和简注。
2. `_上下文快照.md` 的最后动作、下一步、未解决问题、切换原因。
3. `_上下文快照.md` 的 `## 6. 关联会话` 追加当前会话行。
4. `索引.md` 的 status、Next Action、关键产出、时间线日志。
5. `git add` 当前 Topic 目录 + 仪表盘 + log，`git commit -m "🧊 冻结: <Topic>"`。未提交的冻结不受 git 保护。

## 恢复

恢复时先读 `_上下文快照.md`，向林峰汇报：

1. 上次做到哪里。
2. 接下来该做什么。
3. 之前卡在哪里。

确认后再切回 Doing。

## 关闭

Topic 工作目的达成时：

1. 全量更新快照。
2. `_上下文快照.md` 的 `## 6. 关联会话` 追加最终会话行。
3. `索引.md` 改为 `pending_extraction`，并同步 Next Action、关键产出、时间线日志。
4. 仪表盘移入待萃取区，并保持 Topic 行状态为 `[📋 待萃取]`。
5. `git add` 当前 Topic 目录 + 仪表盘 + log + 知识库变更，`git commit -m "✅ 关闭: <Topic>"`。未提交的关闭不受 git 保护。
6. 不自动萃取，等林峰主动触发。
