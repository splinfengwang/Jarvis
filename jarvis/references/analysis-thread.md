# Analysis Thread Lifecycle

## 目的

分析型工作的本质是逐步深化的认知过程——有方向但不预设终点，随分析深度不断收敛。本文档定义分析线程的状态管理和追踪规则。

## 状态值

| 机器值 | 展示值 | 含义 |
|---|---|---|
| `discussing` | `[💬 讨论中]` | 正在分析，方向尚未收敛 |
| `proposed` | `[📝 提案]` | 分析结论已形成，等待确认 |
| `confirmed` | `[✅ 已确认]` | 林峰认可，可作为后续依据 |
| `abandoned` | `[❌ 已废弃]` | 方向放弃 |

## 状态流转

| 变更 | 判定方式 |
|---|---|
| 讨论中 → 提案 | 审计清单已输出、根因已归纳、方案已成形 |
| 提案 → 已确认 | 方案定稿，林峰认可 |
| 提案 → 讨论中 | 回炉，继续分析 |
| 任意态 → 已废弃 | 方向放弃 |

流转规则：
- 贾维斯提议状态变更，林峰确认后执行
- `discussing` 是默认初始状态
- `abandoned` 不删除文档，只标记状态

## related 双链规范

分析线程内各文档通过 frontmatter `related` 字段维护先后关系和依赖：

```yaml
related: [[前序分析文档]] [[并行对比文档]]
```

- `related` 只列出与当前文档有直接分析关系的文档
- 不把整个 Topic 的所有文件都塞进去
- 只写已存在的文档，不写悬空双链

## 版本管理

分析文档 frontmatter 中维护 `version` 字段：

```yaml
version: 1
```

- 每次重大修改（改变结论或增加新分析维度）时递增
- 小修（纠错别字、格式调整）不递增

## 载体

分析文档统一存放在 `业务/<域>/`，通过 `related` 双链形成分析网络。状态写入 frontmatter `status` 字段。
