# 知识库模型

> 参考文档。定义 L1-L4/F 五层知识分层、A/B/C 术语分类、术语生命周期、入账规则、目录职责规范。
> 由 knowledge-extract / knowledge-ingest / knowledge-feedback 按需引用。

## 读取时机

以下场景按需读取：
- 执行 jarvis-knowledge-extract（需理解知识分层和证据类型）
- 执行 jarvis-knowledge-ingest（需理解入库规则和载体规范）
- 执行 jarvis-knowledge-feedback（需理解术语生命周期）
- 用户询问知识库结构或术语管理规则

## 知识分层：L1-L4/F 五层模型

### L1 — 术语/概念

一个业务概念是什么。示例：K-S100Pro、FeCO₂、无创通气。

每条 L1 术语写成独立文件，存放在 `知识库/术语/`。

### L2 — 关系与规则

概念之间怎么协作、系统怎么运作。示例：K-S100Pro 是执行端，DPAP20 是治疗端。

L2 不独立建文件，嵌入在分析文档和项目文档中，通过双链与 L1 链接。

### L3 — 判断与决策

为什么这样设计、选择了什么、放弃了什么。示例：K-S 用 EtCO₂ 做过程控制但评定用 FeCO₂。

L3 嵌入分析文档和项目文档中，不独立建文件。

### L4 — 待验证/未解决

已知但还没答案的问题。示例：锚定差值在治疗过程中是否稳定？

L4 嵌入分析文档或 Topic 的未解决问题区，不独立建文件。

### F — 文件处理产物

对源文件（PDF/PPT/图片/文档）的结构化整理。示例：产品能力清单、模块表、流程表。

F 层产物存放在 `业务/<域>/`，按业务文档格式命名，通过 frontmatter 的 `quality`（📝描述级/🔍对比级/🧠归纳级）和 `source_files` 字段标记来源。

**F 层产物不能整体入库**，必须拆分为：OCR/文档解析原文摘取、Agent 对源材料的结构化整理、Agent 的解释归纳推断、用户的纠正或确认。

## L1 术语三级分类

| 类别 | 名称 | 入库标准 | 示例 |
|------|------|------|------|
| A | 平台业务领域模型术语 | ⭐ 最优先。入库前需确认内容。存放于 `知识库/术语/` | 医护协作端、个案协作端 |
| B | 专业医学知识术语 | 集中积累在 `知识库/术语/`，由入账规则管理 | FeCO₂、无创通气 |
| C | 外部系统产物 | **不入库为独立术语**，记录在对应业务域主文档中。若同一产物在 ≥2 个 Topic 中被引用，提案升级为 A 类 | HIS 收费码体系 |

### 判定流程

1. 先问"这是平台自己的概念，还是外部系统的产物？"
   - 外部产物 → C 类，入主文档，不入术语体系
   - 平台概念 → 进入下一步

2. 再问"这是 Agent 推理需要的基础概念，还是纯领域知识？"
   - 前者 → A 类（平台业务术语）
   - 后者 → B 类（专业知识术语）

3. 设计工具产物直接排除，不入库。

### A/B 边界规则

当术语同时具备 A 和 B 属性时，以"Agent 推理的主要引用场景"为准：

- 用于理解平台功能/流程/角色 → A 类处理，医学属性作为 tags 标注
- 纯粹用于医学证据检索 → B 类处理

## 术语原子化与载体

### 原子化原则

- L1 术语写为独立笔记，一条术语一个文件，自包含
- 通过 `[[双链]]` 建立引用关系
- A 类术语双链回源至业务域主文档；B 类术语双链至引用它的各分析文档
- L2-L4 嵌入分析文档和项目文档中，通过双链与 L1 链接
- 知识库主体是 Obsidian 双链网络，不是文件夹树

### 术语笔记模板

```markdown
---
tags: [术语, <业务标签>]
domain:
  - <业务域>
status: 待确认
related: [[术语A]] [[术语B]] [[术语C]]
---

# <术语名>

**一句话定义**：（这条术语是什么）

**属于**：<业务域>

**相关术语**：
- [[术语A]] — （关系说明）
```

### 术语文件规范

- **related 字段**：多个术语用空格分隔，写 `[[术语名]]`。只有跨目录引用时才用完整路径+别名 `[[完整路径|显示名]]`
- **tags 字段**：A 类术语必须加业务标签，如 `[术语, 设备]`。B 类加 `[术语, 医学]`。不写 `[术语]` 单标签
- **domain 字段**：最多 2 个 domain。第 3 个必须确认是否属于已有 domain 的子分类
- **正文结构**：按术语类型选用章节

| 术语类型 | 推荐章节 |
|------|------|
| 设备类 | 关键属性、核心参数、在治疗流程中的角色 |
| 指标类 | 获取方式、在滴定/治疗中的角色、与相关指标对比 |
| 流程/概念类 | 适用场景、结构、核心规则、使用示例 |
| 平台类 | 流程、关键属性 |

正文不写 L3 判断和 L4 待验证问题——这些放在分析文档或 Topic 中，通过双链引用回术语。

### 悬空双链处理

引用的目标文件不存在时，**不写悬空双链**。在对话中告知用户该术语尚未创建。若用户当场要求创建则立即创建并补链；若未创建，将该双链记入仪表盘跟进事项区（如"补链：FeCO₂ → EtCO₂"），下次会话启动时提醒。

## 术语生命周期

### 批量确认

用户确认术语时支持三种模式：

| 模式 | 触发方式 | 处理 |
|------|------|------|
| 全部确认 | "全部确认" / "都确认" | 批量将所有 `待确认` 改为 `已确认` |
| 指定确认 | "1、3、5 确认" | 仅修改被点名的术语 |
| 逐条确认 | 用户逐条审核内容后确认 | 逐个修改，同时可修正定义 |

### 术语索引同步

新建术语后必须更新 `知识库/术语/术语索引.md`，在对应 domain 分组下追加条目。索引中每条术语标注 status（`待确认` / `已确认`）。

### 术语健康检查

- **触发时机**：用户询问全局近况时，扫描全部 L1 术语
- **筛选条件**：`status: 待确认` 且创建超过 2 周未更新
- **提醒方式**：有新增超标术语且跟进区尚无记录时，写入仪表盘跟进事项区

### 孤立页检测

扫描 `知识库/术语/` 和 `业务/` 下所有文件的双链引用，标记没有任何文件引用它的术语页。发现孤立页时告知用户，由用户决定是补充引用还是删除。

执行方式：grep 所有文件的 `[[` 引用，对比术语文件列表，找出入链为零的页面。

## 入账规则

### 触发时机

对话中出现以下信息时，主动识别并提议入账：

- 新的业务术语或概念定义
- 概念之间的关系或业务规则
- 设计决策和判断依据
- 明确提出的待验证问题

入账前向用户确认。若用户说"先不记"，立即跳过。

### 批量提取

批量提取术语时，先生成提取清单（格式同知识萃取清单），用户逐条确认后再写入。**禁止跳过确认直接入库。**

### 新建术语 status 强制规则

新建术语的 `status` 只能是 `待确认`。不存在"创建时直接确认"的路径。术语经用户审核内容后口头确认，才能改为 `已确认`。

## 载体总览

| 知识层 | 存放位置 | 载体形式 |
|------|------|------|
| L1 术语 | `知识库/术语/` + `术语索引.md` 按 domain 分组导航 | 独立 `.md` 文件 |
| L2-L4 | `业务/<域>/` 或 `platform-ops/topics/` 的分析文档中 | 嵌入分析文档，通过双链与 L1 链接 |
| F 文件处理产物 | `业务/<域>/` | 独立 `.md` 文件，frontmatter 含 `quality` + `source_files` |

## 索引缺口维护

知识检索降级（L3/L4）时，agent 会记录缺口到 `wiki索引.md` 末尾的"索引缺口"表。以下是维护规则。

### 三种缺口类型

| 类型 | 原因 | 处理 |
|------|------|------|
| 知识库空白 | 项目从未涉及该知识 | 用户决定是否新建 Topic/术语 |
| 索引遗漏 | 文件存在但 wiki索引 或 术语索引 未收录 | 立即补入索引 |
| 链接断裂 | wiki索引 中的链接指向不存在的文件 | 立即修复链接 |

### 清理时机

- **链接断裂**：发现时立即修复，不等待
- **索引遗漏**：在以下时机补入索引——
  - Topic 关闭前
  - 知识萃取完成后
  - 每次新建 wiki 页面或术语后
- **知识库空白**：在用户询问近况时提醒，由用户决定

### 仪表盘联动

`platform-ops/仪表盘.md` 中有一行：

```
> 📇 wiki索引: N 个缺口未处理 → 见 [[知识库/wiki索引#索引缺口]]
```

每次用户询问"最近怎么样"时，agent 扫描 wiki索引.md 的索引缺口表，统计未处理数，更新此行。N > 0 时主动提醒用户。

## 目录规格：每个文件夹的职责

### 知识库/术语/

| 维度 | 内容 |
|------|------|
| **存什么** | L1 术语笔记，一条术语一个 `.md` 文件。自包含的定义：是什么、属于哪个 domain、相关术语、关键属性。不包含 L3 判断和 L4 问题 |
| **命名** | `<术语名>.md` |
| **创建者** | `jarvis-knowledge-ingest`（确认入库时写入） |
| **消费者** | `jarvis-knowledge-extract`（对照已有术语）、检索 L1（按术语查定义）、所有需要引用术语的场景 |
| **维护** | `jarvis-knowledge-feedback`（用户纠正时更新 status/内容）。新建术语 status 只能是 `待确认` |
| **双链** | related 字段简写 `[[术语名]]`。A 类术语双链回业务域主文档；B 类双链至分析文档 |
| **同步** | 新建后必须更新 `知识库/术语/术语索引.md` |

### 知识库/术语/术语索引.md

| 维度 | 内容 |
|------|------|
| **存什么** | 全部 L1 术语按 domain 分组导航。每个术语一行：`[[术语名]] — 一句话摘要` |
| **创建者** | `jarvis init` 时生成骨架；`jarvis-knowledge-ingest` 入库新术语时追加 |
| **消费者** | 检索 L1（按 domain 浏览术语体系）、术语健康检查、孤立页检测 |
| **维护** | 新建术语必须同步追加；术语状态变更时更新 status 标记 |

### 知识库/wiki索引.md

| 维度 | 内容 |
|------|------|
| **存什么** | 全知识库扁平导航。覆盖 L1 术语、L2-L4 分析文档、F 文件产物、决策记录。每行：`[[路径|显示名]] + 一句话摘要 + 状态`。末尾有 `## 索引缺口` 表 |
| **创建者** | `jarvis init` 时生成骨架；`jarvis-knowledge-ingest` 入库时追加；检索降级时追加缺口 |
| **消费者** | 检索 L1（总入口，查"有什么知识"）、近况查询（缺口统计） |
| **维护** | **强制同步**：任何新建/迁移 wiki 页面必须同步更新。定期对照（7 天）：扫描 `术语/` 和 `业务/` 补遗漏。Topic 关闭+知识萃取后更新 |

### 业务/<域>/

| 维度 | 内容 |
|------|------|
| **存什么** | L2 关系/规则、L3 判断/决策、L4 待验证问题、F 文件处理产物。嵌入分析文档中，通过双链与 L1 术语链接 |
| **命名** | `YYYY-MM-DD <域>-<主题>.md`。F 产物同格式，通过 frontmatter 的 `quality`（📝/🔍/🧠）+ `source_files` 区分 |
| **创建者** | ① 新建文件时优先判断：脱离 Topic 后还有人需要读到它吗？是 → 直接创建在 `业务/<域>/`，Topic 中只留双链引用。否 → 创建在 Topic 目录 ② `jarvis-knowledge-ingest` G 组独立存放 ③ `jarvis-file-process` 落文档（F 产物）④ 直接在知识库中创建 |
| **消费者** | 检索 L1（查分析文档）、新工作命题（了解已有分析结论）、知识萃取对照（查重复/冲突）|
| **维护** | frontmatter 必须含 `domain`、`status`、`version`、`related`。status 流转：讨论中→提案→已确认→已废弃。废弃不删除 |
| **从 Topic 独立存放** | G 组确认后三步：① 复制到业务/<域>/，改名补 frontmatter → ② 更新 wiki索引 → ③ 原文件改名 _archived_原文件名.md。不修双链，不删原文件。wiki索引是唯一权威入口 |

### platform-ops/仪表盘.md

| 维度 | 内容 |
|------|------|
| **存什么** | 工作状态总视图。活跃 Topic、待萃取 Topic、已归档 Topic、跟进事项、待拍板、wiki索引缺口引用 |
| **创建者** | `jarvis init` 时生成骨架；`jarvis-topic-create`/`freeze`/`resume`/`close` 更新状态行 |
| **消费者** | `jarvis-status`（近况查询）、`jarvis-topic-resume`（恢复 Topic）、检索 L1 入口之一 |
| **维护** | 只 Edit 追加行，禁止整页重写。Topic 状态变更时即时更新。更新后 Read 验证格式 |

### platform-ops/log.md

| 维度 | 内容 |
|------|------|
| **存什么** | 操作日志。ingest/query/lint/install 等操作每条一行 `## [YYYY-MM-DD] 操作类型 | 简述` |
| **创建者** | `jarvis init` 时初始化；各 skill 执行操作后追加 |
| **消费者** | 审阅记录、问题追溯 |
| **维护** | append-only，不修改历史行 |

### platform-ops/topics/<Topic>/（根目录）

| 维度 | 内容 |
|------|------|
| **存什么** | 四件套骨架文件（索引.md / _上下文快照.md / _准入检查单.md / 讨论记录.md）+ 元数据文件（_萃取清单.md / _evidence-pack.json 等）。**不存业务产出文件**。 |
| **命名** | 目录：`YYYYMMDD_主题简称`。骨架和元数据文件名固定。 |
| **创建者** | 骨架：`jarvis-topic-create`。元数据：各 skill 运行时写入。 |
| **消费者** | `jarvis-topic-resume`（恢复上下文）、`jarvis-topic-freeze` / `jarvis-topic-close`（状态同步）、`jarvis-knowledge-extract`（读取 _萃取清单 等元数据）。 |
| **维护** | 冻结/关闭时由对应 skill 更新快照和索引。萃取完成 → Topic ⚪Done 后目录保留。 |
| **归档** | 不变更。 |

### platform-ops/topics/<Topic>/定稿/

| 维度 | 内容 |
|------|------|
| **存什么** | 讨论确认后具备独立价值的最终产出文档：方案终稿、设计说明、Roundtable 审查报告、分析报告等。萃取时的**首选扫描目录**。 |
| **命名** | `<YYYYMMDD>-<主题>-<文档类型>.md`（如 `20260605-适老化改造-架构总览.md`）。不加 `_` 前缀。二级子目录无例外，萃取后脱离目录仍可独立理解。详见 [topic-lifecycle#文件命名规范](topic-lifecycle.md)。 |
| **创建者** | Topic 讨论过程中用户确认后 agent 手动创建/移入。 |
| **消费者** | `jarvis-knowledge-extract`（G 组首要候选来源）。`jarvis-topic-resume`（恢复时定位关键产出）。 |
| **维护** | 文件确认定稿后移入此目录；旧的定稿版本若被取代则移入 `过程稿/`，此目录始终保持最新。 |
| **归档** | 萃取后已迁移到 `业务/` 的文件 → 原文件改名 `_archived_原文件名.md`（不删，不修双链）。未迁移的文件保留。 |

### platform-ops/topics/<Topic>/过程稿/

| 维度 | 内容 |
|------|------|
| **存什么** | 讨论中产生的中间版本、被后续版本取代的旧版、废弃方案、讨论纪要草稿等。保留作为迭代审计记录。 |
| **命名** | `<YYYYMMDD>-<主题>-<文档类型>-v<N.M>.md`（如 `20260608-用户积分体系-设计方案-v0.1.md`）。多子主题时按子主题建二级目录。详见 [topic-lifecycle#文件命名规范](topic-lifecycle.md)。 |
| **创建者** | Topic 讨论过程中 agent 手动创建（初稿、迭代稿）。 |
| **消费者** | `jarvis-topic-resume`（了解迭代历史）。`jarvis-knowledge-extract`：默认**跳过**，除非用户明确要求。 |
| **维护** | 新版本产生后旧版本移入。废弃方向/方案移入。不需要清理——保留完整迭代痕迹。 |
| **归档** | 不萃取。Topic ⚪Done 后随目录保留。 |

### platform-ops/topics/<Topic>/参考资料/

| 维度 | 内容 |
|------|------|
| **存什么** | 从外部进入 Topic 的原始材料：用户拖入的 PDF/PPT/Word/图片、网页爬取原文、截图等。**只读，agent 不应覆写**。 |
| **命名** | 保持原始文件名或略加描述前缀。来源多时按类型建子目录。详见 [topic-lifecycle#文件命名规范](topic-lifecycle.md)。 |
| **创建者** | 用户手动放置，或 Agent 在爬取/文件处理后放入。 |
| **消费者** | `jarvis-knowledge-extract`（仅作溯源引用，不入库）。讨论过程中按需读取作为上下文。 |
| **维护** | 无需维护。纯静态参考。 |
| **归档** | 不萃取。外部参考素材若具长期参考价值 → G 组建议归档到 `知识资产/`。 |

### 知识资产/
