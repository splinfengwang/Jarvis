## 萃取流程（5 步）

### 第 1 步：定位源记录

读取 Topic 的 `_上下文快照.md` → 找到"关联会话"表 → 按工具和日期定位 JSONL 文件。

- JSONL 路径明确 → 标记 `✅`
- JSONL 路径为 `待确认` → 用 `locate_session_jsonl.py --tool claude-code --cwd <项目路径> --date <日期>` 手动搜索，找到候选后标 `⚠️ 候选路径需确认`
- 文件缺失 → 标 `⚠️ 会话文件缺失`，不阻塞其他来源
- 快照中无关联会话 → 声明"缺失会话记录"，仅从 Topic 文件萃取
- 快照距今 > 7 天 → 提醒用户"快照可能过时，是否先更新 Topic 状态再萃取"

**来源清单格式：**

```
来源 1: [工具] [会话ID] → JSONL路径 — ✅/⚠️
来源 2: Topic 文件（索引/快照/讨论记录） — ✅
```

---

### 第 1.5 步：扫描 Topic 产出文件

扫描 Topic 目录下的非骨架文件，分优先级处理。

**扫描规则（按目录区分）：**

| 目录 | 优先级 | 处理方式 |
|------|:------:|------|
| `定稿/` | 🥇 最高 | 全部进入 G 组扫描。独立可复用的完整文档建议迁移到 `业务/<域>/`。 |
| `过程稿/` | 🥉 默认跳过 | 仅保留作为迭代审计记录。除非用户明确要求"把过程稿也萃取一下"。 |
| `参考资料/` | ❌ 不入库 | 仅作溯源引用。外部素材具长期参考价值 → G 组建议归档到 `知识资产/`。 |
| 根目录（非骨架文件） | 🥈 按原有逻辑 | 判断是否独立可复用 → 是则进 G 组，否则留在 Topic。 |

**定稿/ 和根目录下文件分两路处理：**

**完整文档 → 放入 G 组：** 独立成篇的分析/设计/方案文档，本身是完整知识，不值得拆碎。
- 独立可复用（脱离 Topic 上下文仍有价值）→ G 组，建议独立存放到 `业务/<域>/`
- 仅本次 Topic 有意义 → 留在 Topic
- 外部参考素材 → G 组，建议归档到 `知识资产/`

**文档中的碎片知识 → 进入第 2 步：** 例如文档中提到的术语定义、规则判断、决策理由。和其他证据块一起进第 2 步抽取 → A/B/C/D 组。

G 组结构见第 5 步。

---

### 第 2 步：外部来源结构化

**不是可选的。** Topic 中存在网页链接、Confluence 页面、图片、PDF 时——必须先执行这一步，再进证据块抽取。

参考素材中的字段名、术语、数值 → 照搬原文。不替换为近义词、不四舍五入数值、不自创"更好的表述"。除非原文用语和已确认术语冲突 → 以术语为准并标注差异。

对每个外部来源，先判断：

> 能从中提取至少一条候选知识（L1/L2/L3/L4）吗？
>   能 → 全文结构化整理 → 从中抽取证据块
>   不能 → 📎 摘要概括 + 保留链接 → 直接放入 E 组

**全文整理** — 复杂内容，含参数/定义/规则：
```
## 结构化整理: <来源名称>
> 来源类型: 网页/Confluence/图片/PDF
> 来源: <URL/页面ID/文件路径>
> 信任级: 网页可回访/Confluence原文/OCR需回看原图

[整理后的文本，标注关键数据点]
```

**摘要引用** — 简单/参考内容：
```
> 📎 参考: <标题/URL> — <一段摘要> — 原文: <链接>
```

### 第 3 步：抽取证据块

| 证据类型 | 来源 | 示例 |
|---------|------|------|
| 用户定义 | 用户对术语/角色/规则的明确描述 | "K-S100Pro 是 S 系列的呼吸管理仪" |
| 用户决策 | 用户对范围/方案/优先级的拍板 | "PPT 只讲氧疗，不讲无创通气" |
| 用户纠正 | 用户指出 Agent 理解错误或补充正确说法 | "不是 K-S100，是 K-S100Pro" |
| 原始文件 | PDF/PPT/文档等路径或消息附件 | "详见 宣传彩页-v0.7.pptx" |
| OCR/文档解析 | OCR、视觉识别、文档解析工具的原始 tool_result | OCR 输出的设备参数表 |
| Agent 结构化整理 | Agent 对文件/流程/方案的结构化摘要 | "压力滴定流程分为 5 个阶段" |
| Agent 推断/归纳 | Agent 基于多个材料做出的判断或趋势 | "基于 FeCO₂ 和 EtCO₂ 的使用差异，推断..." |
| Topic 记录 | 索引、快照、讨论记录、阶段产出 | 讨论记录中的方案对比表 |

**执行：** 对每条证据，标注：
```
[证据] 类型: [用户定义/用户决策/…]
       内容: [一句话]
       来源: [文件路径 + 消息位置]
```

---

### 第 5 步：补齐检查

> 执行时机：第 4 步（生成候选）之后。检查每个候选知识是否完整。

对每条候选检查 5 个维度：

| 维度 | 查什么 | L1 术语强制 | L2/L3 |
|------|--------|:--:|:--:|
| 定义 | 有没有说清楚"是什么" | ✅ 必须有 | 建议 |
| 分类 | 属于哪个 domain | ✅ 必须有 | 建议 |
| 关系 | 和哪些概念相关 | 建议 | 建议 |
| 决策理由 | 为什么这样选（仅 L3 判断） | — | 建议 |
| 适用范围 | 什么条件下生效 | 建议 | 建议 |

**输出方式：不单独成表。** 检查结果合并到确认清单——每个候选行增加 `⚠️ 待补充` 列。

- 5 维度全部有 → 列值: `—`
- 有缺口 → 写缺口摘要（如"缺定义; 缺domain"），附措辞建议
- 措辞建议保留原意，不加新信息。原表述已清晰 → 不写建议

### 第 4 步：生成候选知识

将证据块转化为候选知识。每条候选**必须写成可判断真假的句子**。禁止"讨论了 XX""涉及 XX"。

| 类型 | 含义 | 示例 |
|------|------|------|
| L1 术语 | 一个业务概念是什么 | "K-S100Pro 是 S 系列呼吸管理仪，旁流 EtCO₂ 自动监测" |
| L2 关系/规则 | 概念之间怎么协作 | "FeCO₂ 用于结果评定，EtCO₂ 仅用于过程控制" |
| L3 判断/决策 | 为什么这样选，放弃什么 | "PPT 只讲氧疗，不讲无创通气——因为受众是基层医生" |
| L4 待验证问题 | 重要但尚无答案 | "FeCO₂ 与 PaCO₂ 的锚定差值在长期治疗中是否稳定？" |
| F 文件处理产物 | 对源文件的结构化整理 | 产品能力清单、模块表、流程表 |

**执行：** 对每条候选，填写。同时扫描候选文本和证据块，识别其中出现的已有术语名称（在术语索引中存在的），自动列入 `related` 建议。

```
[候选] 类型: L1/L2/L3/L4/F
       我们的理解: [一句可判真假的判断句]
       准备用法: [以后在哪类任务中用]
       related: [[已有术语A]] [[已有术语B]]
```

---

### 第 6 步：已有知识对照

对每条候选，搜索 `知识库/术语/术语索引.md` 和 `知识库/wiki索引.md`，检查是否与已有条目重合。

| 对照结果 | 标记 | 放入 | 确认后 |
|---------|------|------|-------|
| 文件名完全相同 | `🔄 重复` | D 组 | 不入库，注明"已有 [[条目名]]" |
| 概念高度相关但文件名不同 | `⚠️ 可能重复` | C 组 | 用户判断是否合并。例："呼吸管理仪" vs "呼吸机" |
| 与已确认知识矛盾 | `❓存疑` | C 组 | 用户裁决。旧条目可能废弃。例：候选中步长 1.0，已确认 0.5 |
| 候选补充了已有知识的细节、例外、子规则 | `✅ 互补` | A 组 | 新条目入库 + 旧条目正文追加。例：已有"上调 2%→+0.5"，候选补充"持续 5min" |
| 候选引用了已有术语，但已有术语未反链 | `🔗 应关联` | A 组 | 新条目入库 + 旧条目 related 追加。例：候选涉及 SpO₂，但 SpO₂.md 的 related 没包含此候选 |
| 无重合 | `🆕 新知识` | 正常流程 | 正常入库 |

同时也检查本次萃取中不同来源对同一主题的说法是否一致。发现 OCR 参数 vs 用户原话不一致、或两个会话之间的说法变化 → 标 `❓存疑` → C 组。

**只标记，不决策。** 不自动合并、不自动拒绝、不自动纠正。用户拥有最终决策权。

---

### 第 7 步：信任分级 + 确认清单

根据证据来源分配信任级别，决定建议动作：

| 来源 | 信任级别 | 入库规则 |
|------|---------|---------|
| 用户明确确认/纠正 | 最高 | 可作为确认依据 |
| 原始文件可核对 | 一手证据 | 可入候选，关键参数待确认 |
| OCR/文档解析 tool_result | 高优先 | 优先于 Agent 二次总结，关键数字需回看原文 |
| Agent 结构化整理 | 候选整理 | 可入清单，不直接当事实 |
| Agent 推断/归纳 | 候选判断 | 必须标注推断，需确认 |
| 无法回溯来源 | 不可信 | **不入库** |

**缺字段不放过：** 没有证据来源 → 不进入清单；准备用法说不清 → 不入库；涉及医学参数 → 默认待验证；只有 Agent 推断 → 不能建议直接确认。

---

### 第 8 步：输出确认清单

清单分 **A/B/C/D/E 五组**，组名就是用户的确认指令。

```
# 萃取确认清单: <Topic 名称>

> 来源 Topic: [[路径]]
> 来源清单: [第 1 步的结果]

## A. 可快速确认
适合批量确认的描述型知识。来自用户明确说法、源文件或 OCR，不含重大判断。含 `✅ 互补` 和 `🔗 应关联`。

| # | 我们的理解 | 类型 | 证据来源 | 标记 | ⚠️ 待补充 | 建议动作 |
|---|---|---|---|---|---|---|
| D1 | … | 描述 | 源文件 + OCR | — | — | 建议确认 |
| D2 | 补充已有规则的细节 | L2 | 用户决策 | ✅ 互补 → [[已有条目]] | — | 建议确认 |
| D3 | 引用已有术语 | L1 | 用户定义 | 🔗 应关联 → [[已有术语]] | 缺定义; 缺domain | 建议确认+补充 |

## B. 需要拍板
涉及范围边界、优先级、产品判断。

| # | 我们的理解 | 影响范围 | 证据来源 | 需要判断 |
|---|---|---|---|---|---|
| J1 | … | … | 用户消息 + Agent 归纳 | 是否作为默认判断 |

## C. 需要验证或裁决
涉医学/安全参数、来源不足、与已有知识矛盾（`❓存疑`）、近名条目需判断合并（`⚠️ 可能重复`）。

| # | 我们的理解 | 风险/疑问 | 证据来源 | 当前处理 |
|---|---|---|---|---|---|
| M1 | … | 医学参数 | OCR/文献 | 待医学验证 |
| C1 | … | 与已确认 [[术语X]] 矛盾 | Agent 推断 vs 已确认 | 用户裁决 | — |

## D. 建议不入库
过程想法、废弃方案、与已有条目重复（`🔄 重复`）。

| # | 内容 | 原因 |
|---|---|---|
| X1 | … | 已被后续讨论废弃 |
| R1 | … | `🔄` 已有 [[条目名]] |

## E. 文件处理产物（F 层）

OCR/文档解析的结构化整理。确认的是"整理有没有遗漏/误读"。

| # | 原始文件 | 已找到的处理产物 | 处理质量 | 建议动作 |
|---|---|---|---|---|---|
| F1 | … | OCR + Agent 结构化整理 | 📝 描述级 | 建议确认后入库 |

确认后 → 写入 `业务/<域>/`，按 F 层规范标记 frontmatter（`quality` + `source_files`）。

## G. Topic 完整文档

第 1.5 步扫描到的独立成篇文档。确认的是"要不要独立存放"。

| # | 文件 | 建议目标 | 建议动作 |
|---|---|---|---|
| G1 | 呼吸机型号定位对比.md | 业务/产品宣传/ | ✅ 建议独立存放 |
| G2 | 参考彩页_BMC.md | 知识资产/ | 📦 建议归档 |

确认后 → 复制到 `业务/<域>/` → 更新 wiki索引 → 原文件改名 `_archived_原文件名.md`（不删，不修双链）。

> **可用确认指令**: A 组全确认 / B1 改成… / C 组全部待验证 / D 组不入库 / E 组确认入库 / G1 独立存放 / G2 归档 / 全部描述级确认，判断类先待确认
> ⚠️ 入库后需同步: wiki索引（新增条目/更新状态）、术语索引（L1 术语按 domain 追加）



**输出后等待用户确认。不自动入库。** 用户确认后 → `jarvis-knowledge-ingest`。

## 硬性规则

| 规则 | 违反后果 |
|------|---------|
| 候选必须写成可判真假的句子 | "讨论了 XX" → 不合格 |
| 没有证据来源 → 不进入清单 | 来源缺失的 claim 不入库 |
| Agent 推断/归纳 → 必须标推断，不能建议直接确认 | 推断当事实 → 用户误确认 |
| 医学参数/安全边界 → 默认待验证 | 跳过验证 → 风险 |
| 文件处理产物 → 拆分，不整体入库 | OCR 原文 / Agent 整理 / Agent 推断 必须分开 |
| 输出确认清单后 → 等用户确认，不自动进入入库 | 跳过确认 → 违反 write_level |
